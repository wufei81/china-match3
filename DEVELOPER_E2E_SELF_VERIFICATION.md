## 📊 职责划分

### dev-engineer 的 E2E 责任

**开发级 E2E 自验证 (必须完成才能交付 QA)**:

| 测试类型              | 目的                           | 示例                           |
| --------------------- | ------------------------------ | ------------------------------ |
| **单元测试**          | 验证主逻辑、边界条件、错误语义 | Jest/Pytest 测试通过           |
| **API 集成测试**      | 验证 API 端点工作              | curl 测试 + 实际响应           |
| **冒烟测试 (主路径)** | 验证端到端主流程               | 完整用户旅程跑通               |
| **错误分支测试**      | 验证异常处理                   | 密码错误、字段缺失等           |
| **前端集成** (如适用) | 验证前后端连通                 | 页面调用 API、Token 存储、跳转 |
| **静态检查**          | 验证代码质量                   | lint/build/type check 通过     |

**交付物**: `VALIDATION_LOG.md` (附完整测试输出)

**⛔ 未完成 → qa-engineer 必须拒绝接收！**

---

### qa-engineer 的 E2E 责任

**QA 级独立验证 (接收交付后执行)**:

| 测试类型         | 目的                 | 示例                     |
| ---------------- | -------------------- | ------------------------ |
| **完整回归测试** | 验证所有旧功能未破坏 | 所有 P0/P1 AC 覆盖       |
| **用户场景测试** | 验证真实用户流程     | 多角色场景、边缘用例     |
| **性能测试**     | 验证系统性能         | 负载测试、响应时间       |
| **安全测试**     | 验证安全漏洞         | SQL 注入、越权访问、CORS |
| **兼容性测试**   | 验证多浏览器/设备    | 跨浏览器、响应式         |

**交付物**: `INTEGRATION_TEST_PLAN.md` (附测试报告)

**✅ 有最终否决权**

## 🎯 工作流程

```
┌─────────────────────────────────────────────────────────┐
│  dev-engineer 开发完成                                   │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  开发者 E2E 自验证 (MANDATORY)                            │
│  → 单元测试通过                                         │
│  → API 集成测试通过                                      │
│  → 主路径冒烟测试通过                                   │
│  → 错误分支测试通过                                     │
│  → 填写 VALIDATION_LOG.md                               │
└─────────────────────────────────────────────────────────┘
                           ↓
                  交付给 QA
                           ↓
┌─────────────────────────────────────────────────────────┐
│  qa-engineer 预验证 (Before Accepting)                  │
│  → 检查 VALIDATION_LOG.md 完整性                        │
│  → 验证开发者 E2E 全部通过                               │
│  → ✅ 通过 → 接受交付                                   │
│  → ❌ 失败 → 拒绝交付                                   │
└─────────────────────────────────────────────────────────┘
                           ↓ (如接受)
┌─────────────────────────────────────────────────────────┐
│  QA 独立验证                                             │
│  → 完整回归测试                                         │
│  → 用户场景测试                                         │
│  → 性能/安全测试                                        │
│  → 输出 INTEGRATION_TEST_PLAN.md                        │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  发布决策                                                │
│  → Pass → 推荐发布                                     │
│  → Fail → 打回修复                                     │
└─────────────────────────────────────────────────────────┘
```

## 🚨 关键原则

### 职责分离

| 原则               | 说明                                            |
| ------------------ | ----------------------------------------------- |
| **dev 不自证清白** | dev-engineer 证明"能用"，qa-engineer 证明"好用" |
| **QA 有否决权**    | qa-engineer 有最终发布否决权                    |
| **分层验证**       | 开发级 → QA 级，层层过滤                        |
| **效率优先**       | dev 过滤低级问题，QA 专注深度测试               |

### 禁止行为

| dev-engineer          | qa-engineer          |
| --------------------- | -------------------- |
| ❌ 跳过 E2E 自验证    | ❌ 接受不完整交付    |
| ❌ 只跑单元测试       | ❌ 跳过预验证        |
| ❌ 声称"看起来没问题" | ❌ 依赖 dev 测试结果 |
| ❌ 给自己"通过"签字   | ❌ 放弃否决权        |
