## 📊 核心功能

### 1. 性能测试类型

**目的**: 系统化评估系统性能表现

#### 测试类型对比

| 类型 | 目的 | 场景 | 指标 |
|------|------|------|------|
| **负载测试** | 验证预期负载下性能 | 日常运营 | 响应时间、吞吐量 |
| **压力测试** | 找到系统极限 | 峰值场景 | 崩溃点、恢复能力 |
| **耐力测试** | 长时间运行稳定性 | 持续运营 | 内存泄漏、性能衰减 |
| **尖峰测试** | 突发流量应对 | 秒杀活动 | 弹性伸缩能力 |
| **配置测试** | 最优配置参数 | 性能调优 | 不同配置对比 |

#### 性能指标定义

```markdown
## 性能指标

### 响应时间
- **P50**: 50% 请求的响应时间 (中位数)
- **P95**: 95% 请求的响应时间
- **P99**: 99% 请求的响应时间 (长尾)
- **平均**: 算术平均值 (易受异常值影响)

### 吞吐量
- **QPS**: 每秒查询数
- **TPS**: 每秒事务数
- **并发用户**: 同时在线用户数

### 资源使用
- **CPU 使用率**: 处理器使用百分比
- **内存使用率**: 内存使用百分比
- **磁盘 IO**: 读写速度
- **网络带宽**: 网络吞吐量

### 可用性
- **成功率**: 成功请求/总请求
- **错误率**: 失败请求/总请求
```

#### k6 测试脚本示例

```javascript
// k6 性能测试脚本
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate, Trend } from 'k6/metrics';

// 自定义指标
const errorRate = new Rate('errors');
const loginTime = new Trend('login_time');

export const options = {
  // 阶段式负载
  stages: [
    { duration: '2m', target: 100 },   // 热身到 100 用户
    { duration: '5m', target: 100 },   // 稳定负载
    { duration: '2m', target: 200 },   // 增加到 200 用户
    { duration: '5m', target: 200 },   // 稳定负载
    { duration: '2m', target: 500 },   // 压力测试
    { duration: '5m', target: 500 },   // 稳定负载
    { duration: '2m', target: 0 },     // 冷却
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],  // P95 < 500ms
    http_req_failed: ['rate<0.01'],    // 错误率 < 1%
    errors: ['rate<0.1'],              // 自定义错误率 < 10%
    login_time: ['p(95)<1000'],        // 登录 P95 < 1s
  },
};

export default function () {
  const loginStart = Date.now();
  
  // 登录请求
  const loginRes = http.post('https://api.example.com/login', JSON.stringify({
    email: `user_${__VU}@example.com`,
    password: 'password123'
  }), {
    headers: { 'Content-Type': 'application/json' },
  });
  
  loginTime.add(Date.now() - loginStart);
  
  check(loginRes, {
    'login status is 200': (r) => r.status === 200,
    'login has token': (r) => JSON.parse(r.body).token !== undefined,
  });
  
  errorRate.add(loginRes.status !== 200);
  
  sleep(1);
  
  // 获取用户信息
  const userRes = http.get('https://api.example.com/users/me', {
    headers: { 
      'Authorization': `Bearer ${JSON.parse(loginRes.body).token}`,
    },
  });
  
  check(userRes, {
    'user status is 200': (r) => r.status === 200,
  });
  
  sleep(1);
}
```

### 3. 性能测试流程

**目的**: 规范化性能测试执行流程

#### 测试流程

```markdown
## 性能测试流程

### Phase 1: 规划 (1-2 天)
1. **定义目标**: 性能指标、成功标准
2. **识别场景**: 关键用户旅程
3. **设计用例**: 测试脚本、测试数据
4. **准备环境**: 独立性能环境

### Phase 2: 执行 (1-3 天)
1. **基线测试**: 建立性能基线
2. **负载测试**: 预期负载验证
3. **压力测试**: 极限测试
4. **耐力测试**: 长时间运行

### Phase 3: 分析 (1-2 天)
1. **数据收集**: 指标、日志、追踪
2. **瓶颈识别**: 性能分析
3. **根因分析**: 5 Whys
4. **优化建议**: 具体改进方案

### Phase 4: 验证 (1 天)
1. **优化实施**: 开发修复
2. **回归测试**: 验证改进
3. **报告输出**: 最终报告
```

### 4. 性能分析与优化

**目的**: 识别性能瓶颈并提供优化建议

#### 常见性能问题

| 问题类型 | 症状 | 排查方法 | 优化方案 |
|---------|------|---------|---------|
| **慢查询** | DB 响应慢 | 慢查询日志、EXPLAIN | 添加索引、优化 SQL |
| **N+1 查询** | 大量 DB 请求 | 查询计数监控 | 批量查询、预加载 |
| **缓存未命中** | 缓存命中率低 | 缓存监控 | 优化缓存策略 |
| **内存泄漏** | 内存持续增长 | 堆分析、GC 日志 | 修复泄漏点 |
| **连接泄漏** | 连接池耗尽 | 连接池监控 | 修复连接管理 |
| **锁竞争** | 线程阻塞 | 线程 dump | 优化锁粒度 |

#### 性能分析报告模板

```markdown
# 性能测试报告 - [项目名称]

## 执行摘要
- **测试日期**: 2026-03-26
- **测试工具**: k6
- **并发用户**: 100-1000
- **总请求数**: 1,000,000

## 测试结果
### 响应时间
| 场景 | P50 | P95 | P99 | 目标 | 结果 |
|------|-----|-----|-----|------|------|
| 登录 | 120ms | 250ms | 380ms | <500ms | ✅ |
| 搜索 | 180ms | 350ms | 520ms | <500ms | ⚠️ |
| 下单 | 280ms | 480ms | 720ms | <500ms | ⚠️ |

### 吞吐量
| 并发用户 | QPS | 成功率 |
|---------|-----|-------|
| 100 | 850 | 99.9% |
| 500 | 3200 | 99.5% |
| 1000 | 4500 | 98.2% |

### 资源使用
| 指标 | 平均 | 峰值 | 目标 |
|------|------|------|------|
| CPU | 45% | 78% | <80% |
| 内存 | 62% | 75% | <80% |
| 磁盘 IO | 30% | 55% | <70% |

## 瓶颈分析
### 问题 1: 搜索接口 P99 超标
**现象**: P99 响应时间 520ms > 目标 500ms

**根因**: 
1. 商品搜索缺少索引
2. 分页查询效率低

**建议**:
1. 添加商品名称索引
2. 使用游标分页替代 offset

### 问题 2: 下单接口并发能力不足
**现象**: 并发>500 时错误率上升

**根因**:
1. 数据库连接池过小 (50)
2. 库存扣减锁竞争

**建议**:
1. 增加连接池到 200
2. 使用 Redis 预扣减库存

## 优化计划
| 优化项 | 优先级 | 预计改进 | 负责人 |
|-------|-------|---------|-------|
| 添加搜索索引 | P0 | P95 -30% | [姓名] |
| 增加连接池 | P0 | 并发 +50% | [姓名] |
| Redis 预扣减 | P1 | 并发 +100% | [姓名] |
```
