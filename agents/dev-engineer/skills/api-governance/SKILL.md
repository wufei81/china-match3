### 1. API 设计规范

**目的**: 统一 API 设计风格，降低学习和集成成本

#### RESTful API 设计规范

```markdown
## RESTful API 设计规范

### 基础规范
#### URL 设计
- ✅ 使用名词复数：`/api/users`
- ✅ 小写字母，连字符分隔：`/api/user-profiles`
- ✅ 避免动词：❌ `/api/getUsers` → ✅ `GET /api/users`
- ✅ 资源嵌套不超过 3 层：`/api/users/123/posts`

#### HTTP 方法
| 方法 | 用途 | 幂等 |
|------|------|------|
| GET | 查询资源 | ✅ |
| POST | 创建资源 | ❌ |
| PUT | 全量更新 | ✅ |
| PATCH | 部分更新 | ❌ |
| DELETE | 删除资源 | ✅ |

#### 状态码
| 状态码 | 含义 | 使用场景 |
|-------|------|---------|
| 200 | 成功 | GET/PUT/PATCH成功 |
| 201 | 已创建 | POST 成功 |
| 204 | 无内容 | DELETE 成功 |
| 400 | 请求错误 | 参数验证失败 |
| 401 | 未认证 | Token 缺失/过期 |
| 403 | 无权限 | 权限不足 |
| 404 | 未找到 | 资源不存在 |
| 409 | 冲突 | 资源冲突 |
| 422 | 验证失败 | 业务验证失败 |
| 429 | 限流 | 请求超限 |
| 500 | 服务器错误 | 内部错误 |
```

### 2. API 版本管理

**目的**: 管理 API 演进，保证向后兼容

#### 版本策略对比

| 策略 | 示例 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **URL 路径** | `/api/v1/users` | 清晰、易缓存 | URL 冗长 | 大多数场景 |
| **查询参数** | `/api/users?version=1` | URL 简洁 | 不易缓存 | 内部 API |
| **请求头** | `Accept: application/vnd.api.v1+json` | 符合 REST | 调试困难 | 公开 API |
| **域名** | `v1.api.example.com` | 完全隔离 | 运维复杂 | 大版本变更 |

#### 版本管理规范

```markdown
## API 版本管理

### 版本号规则
- 格式：`v{major}.{minor}.{patch}`
- major: 不兼容变更
- minor: 向后兼容功能
- patch: 向后兼容修复

### 版本生命周期
| 阶段 | 时长 | 说明 |
|------|------|------|
| **Current** | - | 当前版本 |
| **Supported** | 12 个月 | 维护模式，只修复 bug |
| **Deprecated** | 6 个月 | 废弃警告，建议迁移 |
| **Retired** | - | 停止服务 |

### 版本迁移
#### 迁移通知
- 废弃前 6 个月：邮件通知
- 废弃前 3 个月：响应头警告
- 废弃前 1 个月：强提醒

#### 迁移指南
```markdown
# v1 → v2 迁移指南

## 变更概述
[主要变更说明]

## 不兼容变更
| 接口 | v1 | v2 | 迁移方式 |
|------|-----|-----|---------|
| GET /users | 返回 flat 结构 | 返回 JSON:API 格式 | [指南链接] |

## 迁移步骤
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

## 常见问题
[FAQ]
```

### 3. API 评审流程

**目的**: 确保 API 设计质量，避免设计缺陷

#### API 评审清单

```markdown
## API 评审清单

### 设计评审
- [ ] 符合 RESTful 规范
- [ ] 资源命名清晰
- [ ] HTTP 方法使用正确
- [ ] 状态码使用恰当
- [ ] 请求/响应结构合理

### 安全评审
- [ ] 认证机制完善
- [ ] 授权校验正确
- [ ] 输入验证完整
- [ ] 敏感数据脱敏
- [ ] 限流策略配置

### 性能评审
- [ ] 支持分页
- [ ] 支持字段过滤
- [ ] 支持排序
- [ ] 缓存策略合理
- [ ] N+1 查询优化

### 兼容性评审
- [ ] 向后兼容
- [ ] 版本策略明确
- [ ] 废弃计划清晰

### 文档评审
- [ ] API 文档完整
- [ ] 示例代码正确
- [ ] 错误码说明清晰
- [ ] 变更日志更新
```

#### 评审流程

```
1. 设计者提交 RFC
   ↓
2. 自动检查 (Lint)
   ↓
3. 同行评审 (3 人)
   ↓
4. 安全评审
   ↓
5. 架构评审
   ↓
6. 批准/修改
   ↓
7. 实施
   ↓
8. 上线前验证
```

### 4. API 治理流程

**目的**: 建立 API 全生命周期治理机制

#### 治理框架

```markdown
## API 治理框架

### 组织架构
| 角色 | 职责 |
|------|------|
| **API 委员会** | 制定规范、评审重大变更 |
| **API Owner** | 负责 API 设计和维护 |
| **API 用户** | 使用反馈、参与评审 |

### 治理流程
#### 设计阶段
- API 设计规范培训
- 设计模板提供
- 同行评审

#### 开发阶段
- API Lint 检查
- 自动化测试
- Mock 服务

#### 发布阶段
- API 文档生成
- SDK 生成
- 变更通知

#### 运营阶段
- 使用监控
- 性能分析
- 用户反馈

#### 废弃阶段
- 废弃通知
- 迁移支持
- 下线执行

### 度量指标
| 指标 | 目标 | 测量方式 |
|------|------|---------|
| API 可用性 | >99.9% | 监控 |
| API 延迟 (P99) | <500ms | 监控 |
| 文档完整度 | 100% | 检查 |
| 评审覆盖率 | 100% | 流程 |
| 用户满意度 | >4.5/5 | 调研 |
```

## 📋 输出规范

1. **规范统一**: 遵循 RESTful 规范
2. **文档完整**: API 文档、示例、错误码齐全

4. **评审严格**: 所有 API 必须经过评审

## ⚠️ 注意事项

- ⛔ API 设计要考虑向后兼容
- ⛔ 不要随意破坏性变更
- ⛔ 废弃 API 要给足够迁移时间
- ✅ API 是产品，要重视开发者体验
