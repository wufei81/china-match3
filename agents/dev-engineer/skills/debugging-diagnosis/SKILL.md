## 📊 核心功能

### 1. 系统化调试方法论

**目的**: 用科学方法定位问题，避免盲目试错

#### 调试流程

```
1. 复现问题
   - 稳定复现步骤
   - 记录环境信息
   - 收集错误日志

2. 定位范围
   - 二分法缩小范围
   - 确定问题边界
   - 识别影响面

3. 提出假设
   - 基于现象推测原因
   - 列出所有可能
   - 按概率排序

4. 验证假设
   - 设计验证实验
   - 逐一排除
   - 找到根因

5. 修复验证
   - 实施修复
   - 回归测试
   - 监控验证
```

#### 调试策略矩阵

| 策略 | 适用场景 | 方法 |
|------|---------|------|
| **自顶向下** | 系统级问题 | 从入口开始追踪 |
| **自底向上** | 底层 bug | 从问题点向上追溯 |
| **二分法** | 大范围定位 | 逐步缩小一半范围 |
| **差异对比** | 回归问题 | 对比正常/异常差异 |
| **隔离法** | 偶发问题 | 隔离变量逐一测试 |

### 2. 根因分析 (5 Whys)

**目的**: 找到问题根本原因，避免表面修复

#### 5 Whys 方法

```
问题：网站访问慢

1. Why? → 服务器响应慢
2. Why? → 数据库查询慢
3. Why? → 缺少索引
4. Why? → 上线前未做性能测试
5. Why? → 没有性能测试流程

根因：缺少性能测试流程
对策：建立性能测试流程，而非仅仅加索引
```

#### 根因分析模板

```markdown
## 根因分析 - [问题描述]

### 问题描述
- **现象**: [具体问题]
- **影响**: [影响范围]
- **时间**: [发生时间]
- **频率**: [必现/偶发]

### 5 Whys 分析
1. **Why 1**: [问题] → [原因 1]
2. **Why 2**: [原因 1] → [原因 2]
3. **Why 3**: [原因 2] → [原因 3]
4. **Why 4**: [原因 3] → [原因 4]
5. **Why 5**: [原因 4] → [根因]

### 根因
**根本原因**: [根因]

** contributing factors**:
- [因素 1]
- [因素 2]

### 对策
| 对策 | 类型 | 负责人 | 完成时间 |
|------|------|-------|---------|
| [短期修复] | 治标 | [姓名] | [时间] |
| [长期预防] | 治本 | [姓名] | [时间] |

### 验证
- [ ] 修复已验证
- [ ] 回归测试通过
- [ ] 监控无异常
- [ ] 流程已更新
```

### 3. 常见故障模式库

**目的**: 基于历史经验快速定位常见问题

#### 后端常见故障

| 故障类型 | 典型症状 | 排查方向 |
|---------|---------|---------|
| **内存泄漏** | 内存持续增长，最终 OOM | 堆分析、GC 日志 |
| **连接泄漏** | 连接池耗尽，请求超时 | 连接池监控、代码审查 |
| **死锁** | 线程阻塞，CPU 低但无响应 | 线程 dump、锁分析 |
| **慢查询** | 接口响应慢，DB CPU 高 | 慢查询日志、执行计划 |
| **缓存穿透** | 缓存失效，DB 压力激增 | 缓存命中率、热 key 分析 |
| **雪崩效应** | 服务级联失败 | 依赖分析、熔断状态 |

#### 前端常见故障

| 故障类型 | 典型症状 | 排查方向 |
|---------|---------|---------|
| **白屏** | 页面空白 | JS 错误、资源加载 |
| **卡顿** | 操作响应慢 | 性能分析、渲染分析 |
| **内存泄漏** | 页面越用越卡 | DevTools Memory |
| **接口失败** | 数据不展示 | 网络请求、CORS |
| **样式错乱** | 布局异常 | 浏览器兼容、CSS 冲突 |

#### 故障排查速查表

```markdown
## 故障排查速查表

### CPU 100%
1. top 定位高 CPU 进程
2. perf 分析热点函数
3. 检查死循环/复杂计算

### 内存 100%
1. 检查内存泄漏
2. 分析堆 dump
3. 检查缓存配置

### 磁盘 100%
1. du 查找大文件
2. 检查日志轮转
3. 清理临时文件

### 网络超时
1. ping/traceroute 检查连通性
2. telnet 检查端口
3. 检查防火墙/安全组

### 服务不可用
1. 检查服务状态
2. 查看错误日志
3. 检查依赖服务
```

## 🔧 使用示例

### 示例 1: API 响应慢排查

**输入**: "API 响应突然变慢，怎么排查"

**输出**:
```markdown
## 排查步骤

### 1. 快速定位
```bash
# 检查系统资源
top -c

# 检查磁盘 IO
iostat -x 1

# 检查网络连接
ss -tuln | grep :8080
```

### 2. 应用层排查
```bash
# 查看应用日志
tail -f app.log | grep -i "slow\|timeout"

# 查看慢查询 (MySQL)
SHOW PROCESSLIST;
```

### 3. 可能原因
| 原因 | 验证方法 | 解决 |
|------|---------|------|
| 慢查询 | 慢查询日志 | 加索引 |
| 连接池耗尽 | 连接池监控 | 调大连接数 |
| 外部依赖慢 | 链路追踪 | 优化依赖 |
| GC 频繁 | GC 日志 | 调优 JVM |
```

### 示例 2: 内存泄漏排查

**输入**: "Node.js 服务内存持续增长"

**输出**:
```markdown
## 内存泄漏排查

### 1. 确认泄漏
```bash
# 监控内存
pm2 monit

# 生成堆快照
node --inspect app.js
# Chrome DevTools 抓取堆快照
```

### 2. 常见原因
| 原因 | 检查点 |
|------|-------|
| 全局变量 | 检查 global 对象 |
| 闭包引用 | 检查长生命周期闭包 |
| 事件监听 | 检查未移除的监听器 |
| 缓存无上限 | 检查缓存策略 |

### 3. 修复建议
- 使用 WeakMap/WeakSet
- 及时移除事件监听
- 缓存设置 TTL 和上限
- 使用内存分析工具定位
```
