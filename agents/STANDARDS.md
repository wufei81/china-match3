# OpenClaw 产研智能体 - 交付优先级共识

> 本文件为 `@workspace/agents` 全链路智能体的**不可违背共识**。每个智能体必须在其职责范围内严格执行此优先级。

## 交付优先级（按序执行，不可颠倒）

| 优先级              | 要求                         | 说明                                                                                                                          |
| ------------------- | ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **1. 实现真实功能** | 功能必须真正可用             | 严禁占位符、空实现、Mock 替代真实 API/DB。表单必须真实提交、接口必须真实读写、按钮必须真实触发。**代码无报错 ≠ 功能已实现**。 |
| **2. 能运行**       | 必须能启动、能执行、无阻塞   | 应用可启动、接口可调用、E2E 关键流程可走通。不得以"构建通过"替代"功能可运行"。                                                |
| **3. 界面美观**     | 在功能可用、可运行前提下追求 | 视觉还原、交互流畅、设计 Token 一致。**不得为追求美观而牺牲功能实现**。                                                       |

## 根因：为何会产出"无报错但无功能"的代码？

1. **AI 优化目标偏差**：模型默认优化"语法正确"（无编译错误），而非"语义正确"（真实行为）
2. **占位符惯性**：生成骨架代码、空 handler、return [] 比实现真实逻辑更"安全"
3. **验收标准模糊**：以"代码无报错""构建通过"为完成依据，未强制 E2E 验证
4. **优先级未显式**：未将"实现真实功能"置于绝对首位

## 各智能体职责对照

| 智能体          | 在交付优先级中的职责                                                  |
| --------------- | --------------------------------------------------------------------- |
| product-manager | AC 必须可 E2E 验证，支持"功能可用"验收                                |
| architect       | 契约必须明确到可实现粒度，支持真实逻辑落地                            |
| ui-ux-designer  | 设计支持功能实现，美观不阻碍开发                                      |
| dev-engineer    | 真实 DB 读写、真实业务逻辑、真实 API 调用、真实表单提交，自验证后交付 |
| qa-engineer     | E2E 功能验证优先，功能不可用为 P0 阻塞                                |
| devops-engineer | 发布门禁含 E2E 通过，不以构建通过为发布依据                           |
| project-manager | 验收含 E2E 功能验证，不以代码无报错为完成依据                         |
| ai-orchestrator | 协调全链路执行此优先级，E2E 未通过不签署发布                          |
