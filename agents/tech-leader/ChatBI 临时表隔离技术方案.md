# ChatBI 临时表隔离技术实施方案

## 一、需求分析

### 1.1 核心需求

- **临时表存储**: 使用 DuckDB/Redis 存储用户查询生成的临时表
- **生产隔离**: 与生产 MySQL 数据库完全隔离，避免相互影响
- **性能保障**: 保证临时表查询性能，支持高并发场景

### 1.2 使用场景

- 用户自定义分析查询生成中间结果表
- 多步骤数据分析的临时存储
- 会话级临时数据缓存
- 复杂查询的结果物化

## 二、技术栈选型

### 2.1 存储层选型对比

| 方案         | 优势                                        | 劣势                       | 适用场景                 |
| ------------ | ------------------------------------------- | -------------------------- | ------------------------ |
| **DuckDB**   | 列式存储、分析查询快、支持 SQL、内存+持久化 | 并发写入有限、分布式支持弱 | 复杂分析查询、中等数据量 |
| **Redis**    | 超高并发、低延迟、数据结构丰富              | 内存成本高、复杂查询弱     | 缓存、简单 KV、高频访问  |
| **混合方案** | 兼顾性能与功能                              | 架构复杂度增加             | 推荐方案                 |

### 2.2 推荐技术栈

```
┌─────────────────────────────────────────────────────────┐
│                    应用层 (ChatBI)                       │
├─────────────────────────────────────────────────────────┤
│              临时表管理服务 (TempTable Service)          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │ 路由决策器   │  │ 生命周期管理 │  │ 监控告警    │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
├─────────────────────────────────────────────────────────┤
│                    存储适配层                             │
│  ┌─────────────────────┐  ┌─────────────────────┐      │
│  │   DuckDB Adapter    │  │   Redis Adapter     │      │
│  │   - 复杂分析查询     │  │   - 热点数据缓存    │      │
│  │   - 中等数据量       │  │   - 高频简单查询    │      │
│  │   - 会话级持久化     │  │   - 元数据索引      │      │
│  └─────────────────────┘  └─────────────────────┘      │
├─────────────────────────────────────────────────────────┤
│                    生产数据库 (MySQL)                    │
│              (与临时表完全隔离)                          │
└─────────────────────────────────────────────────────────┘
```

### 2.3 技术选型决策

| 组件     | 选型                 | 版本 | 理由                           |
| -------- | -------------------- | ---- | ------------------------------ |
| 分析存储 | DuckDB               | 1.0+ | 列式存储、SQL 兼容、嵌入式部署 |
| 缓存存储 | Redis                | 7.0+ | 高并发、RedisJSON 模块         |
| 连接池   | HikariCP             | 5.0+ | 高性能 JDBC 连接池             |
| 序列化   | Protobuf             | 3.x  | 高效二进制序列化               |
| 监控     | Prometheus + Grafana | -    | 标准监控方案                   |

## 三、架构设计

### 3.1 系统架构

```yaml
临时表管理服务架构:
  API 网关层:
    - REST API: /api/v1/temp-tables/*
    - gRPC: 内部服务调用
    - 认证鉴权：JWT + RBAC

  核心服务层:
    路由决策器:
      - 数据量评估：<100MB → Redis, >100MB → DuckDB
      - 查询类型评估：简单 KV → Redis, 复杂 SQL → DuckDB
      - 访问频率评估：高频 → Redis, 低频 → DuckDB

    DuckDB 引擎:
      - 连接池管理：最大 50 连接
      - 文件存储：/data/duckdb/temp/{session_id}/{table_id}.duckdb
      - 内存限制：单实例 4GB
      - 自动清理：TTL 24 小时

    Redis 引擎:
      - 数据结构：Hash(元数据) + Sorted Set(索引) + String(数据)
      - 集群模式：Redis Cluster 3 主 3 从
      - 内存策略：allkeys-lru
      - TTL: 2 小时

  存储层:
    - DuckDB: 本地 SSD 存储
    - Redis: 内存 + AOF 持久化
    - MySQL: 仅存储元数据引用
```

### 3.2 数据流设计

```
用户查询请求
    │
    ▼
┌─────────────────┐
│  查询解析器      │ → 识别临时表操作
└─────────────────┘
    │
    ▼
┌─────────────────┐
│  路由决策器      │ → 选择存储引擎
└─────────────────┘
    │
    ├──────────────┬──────────────┐
    ▼              ▼              ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Redis   │  │ DuckDB  │  │ MySQL   │
│ (热数据) │  │ (分析)  │  │ (元数据)│
└─────────┘  └─────────┘  └─────────┘
    │              │              │
    └──────────────┴──────────────┘
                   │
                   ▼
          ┌─────────────────┐
          │  结果合并返回    │
          └─────────────────┘
```

### 3.3 表结构设计

#### MySQL 元数据表 (生产库)

```sql
CREATE TABLE temp_table_metadata (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    session_id      VARCHAR(64) NOT NULL,
    table_name      VARCHAR(128) NOT NULL,
    storage_type    ENUM('REDIS', 'DUCKDB') NOT NULL,
    storage_key     VARCHAR(256) NOT NULL,  -- Redis Key 或 DuckDB 文件路径
    schema_json     JSON NOT NULL,          -- 表结构定义
    row_count       BIGINT DEFAULT 0,
    size_bytes      BIGINT DEFAULT 0,
    created_at      DATETIME NOT NULL,
    expires_at      DATETIME NOT NULL,
    last_accessed   DATETIME,
    status          ENUM('ACTIVE', 'EXPIRED', 'DELETED') DEFAULT 'ACTIVE',
    INDEX idx_session (session_id),
    INDEX idx_expires (expires_at),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### Redis 数据结构

```
# 元数据
temp:meta:{table_id} = Hash {
    session_id, table_name, schema, created_at, expires_at
}

# 数据 (列式存储)
temp:data:{table_id}:{column_name} = List [values...]

# 索引
temp:index:{table_id} = Sorted Set { row_id -> score }

# 会话表列表
temp:session:{session_id} = Set { table_id1, table_id2, ... }
```

#### DuckDB 表结构

```sql
-- 每个临时表独立 DuckDB 文件
-- 支持标准 SQL DDL/DML
CREATE TABLE temp_data (
    -- 动态列定义，根据查询结果
    column_1 TYPE1,
    column_2 TYPE2,
    ...
);

-- 内置视图用于监控
CREATE VIEW temp_stats AS
SELECT
    count(*) as row_count,
    pg_total_relation_size('temp_data') as size_bytes;
```

## 六、风险控制

### 6.1 技术风险

| 风险            | 概率 | 影响 | 应对措施                                                           |
| --------------- | ---- | ---- | ------------------------------------------------------------------ |
| DuckDB 并发瓶颈 | 中   | 高   | - 限制单实例连接数<br>- 实现请求队列<br>- 热点数据自动迁移到 Redis |
| Redis 内存溢出  | 中   | 高   | - 设置内存上限<br>- 实现 LRU 淘汰<br>- 监控告警 + 自动扩容         |
| 数据一致性问题  | 低   | 高   | - 元数据操作加锁<br>- 实现最终一致性校验<br>- 定期对账             |
| 查询性能不达标  | 中   | 中   | - 建立性能基线<br>- 查询超时控制<br>- 慢查询分析和优化             |

### 6.2 运维风险

| 风险         | 概率 | 影响 | 应对措施                                                           |
| ------------ | ---- | ---- | ------------------------------------------------------------------ |
| 存储空间不足 | 中   | 中   | - 容量监控告警<br>- 自动清理过期数据<br>- 弹性扩容方案             |
| 服务不可用   | 低   | 高   | - 多副本部署<br>- 健康检查 + 自动重启<br>- 降级策略 (返回缓存数据) |
| 数据丢失     | 低   | 高   | - Redis AOF 持久化<br>- DuckDB 定期备份<br>- 元数据 MySQL 主从     |

### 6.3 业务风险

| 风险         | 概率 | 影响 | 应对措施                                                 |
| ------------ | ---- | ---- | -------------------------------------------------------- |
| 用户查询超时 | 中   | 中   | - 查询超时控制 (30s)<br>- 异步查询支持<br>- 进度查询接口 |
| 临时表滥用   | 中   | 低   | - 单用户表数量限制<br>- 单表大小限制<br>- 配额管理       |
