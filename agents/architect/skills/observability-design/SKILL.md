## 🎯 技能概述

为架构师智能体提供可观测性设计能力，包括三大支柱 (指标/日志/追踪)、监控体系、告警设计、SLO 管理。帮助从"被动救火"升级为"主动预防"。

## 📊 核心功能

### 1. 可观测性三大支柱

**目的**: 构建完整的可观测性体系，快速定位和解决问题

#### 三大支柱对比

| 支柱 | 描述 | 工具示例 | 用途 |
|------|------|---------|------|
| **Metrics (指标)** | 数值化系统状态 | Prometheus, Grafana | 趋势分析、告警 |
| **Logging (日志)** | 事件记录 | ELK, Loki | 问题排查、审计 |
| **Tracing (追踪)** | 请求链路追踪 | Jaeger, Zipkin | 性能分析、瓶颈定位 |

#### 指标体系设计

```markdown
## 指标体系设计 - [系统名称]

### 黄金指标 (Four Golden Signals)
| 指标类型 | 描述 | 告警阈值 |
|---------|------|---------|
| **延迟** | 请求处理时间 | P99 > 1s |
| **流量** | QPS/并发数 | 突增/突降 50% |
| **错误** | 错误率 | > 1% |
| **饱和度** | 资源使用率 | CPU/Mem > 80% |

### 业务指标
| 指标 | 定义 | 采集频率 |
|------|------|---------|
| 订单量 | 每分钟订单数 | 1min |
| 支付成功率 | 成功支付/发起支付 | 1min |
| 活跃用户 | DAU/MAU | 1h |
| 转化率 | 下单用户/浏览用户 | 1h |

### 技术指标
| 层级 | 指标 | 采集方式 |
|------|------|---------|
| **基础设施** | CPU/内存/磁盘/网络 | Node Exporter |
| **中间件** | DB QPS/缓存命中率 | Exporter |
| **应用** | JVM/Go Runtime | SDK |
| **业务** | 订单/支付/用户 | 埋点 |
```

#### 日志规范

```markdown
## 日志规范 - [系统名称]

### 日志级别
| 级别 | 使用场景 | 示例 |
|------|---------|------|
| **ERROR** | 错误，需要处理 | 数据库连接失败 |
| **WARN** | 警告，不影响功能 | 重试成功 |
| **INFO** | 重要业务事件 | 用户登录、订单创建 |
| **DEBUG** | 调试信息 | 参数详情 |
| **TRACE** | 详细追踪 | 每步执行 |

### 日志格式 (JSON)
```json
{
  "timestamp": "2026-03-26T13:30:00Z",
  "level": "INFO",
  "service": "order-service",
  "trace_id": "abc123",
  "span_id": "def456",
  "user_id": "user_789",
  "action": "order_created",
  "message": "订单创建成功",
  "data": {
    "order_id": "order_001",
    "amount": 199.00
  }
}
```

### 日志采集策略
- **应用日志**: Filebeat → Kafka → Logstash → ES
- **系统日志**: Journald → Loki
- **访问日志**: Nginx → Filebeat → ES
- **审计日志**: 独立存储，保留 3 年
```

#### 链路追踪设计

```markdown
## 链路追踪设计 - [系统名称]

### 追踪架构
```
应用 (OpenTelemetry SDK)
    ↓
Collector (聚合/采样)
    ↓
Backend (Jaeger/Tempo)
    ↓
UI (查询/可视化)
```

### Span 设计
| Span | 操作 | 标签 |
|------|------|------|
| HTTP Request | method, url, status | http.method, http.status_code |
| DB Query | SQL, 表名 | db.system, db.statement |
| Cache | key, operation | cache.operation, cache.key |
| RPC | service, method | rpc.service, rpc.method |

### 采样策略
- **默认采样**: 10%
- **错误追踪**: 100%
- **慢请求**: 100% (>1s)
- **VIP 用户**: 100%
```

### 2. 监控体系设计

**目的**: 全方位监控系统状态，及时发现问题

#### 监控分层

```
┌─────────────────────────────────┐
│       业务监控 (Business)       │
│   订单/支付/转化/用户行为        │
├─────────────────────────────────┤
│       应用监控 (Application)    │
│   QPS/延迟/错误率/JVM/Go        │
├─────────────────────────────────┤
│       中间件监控 (Middleware)   │
│   MySQL/Redis/MQ/ES             │
├─────────────────────────────────┤
│       基础设施监控 (Infra)      │
│   CPU/内存/磁盘/网络            │
└─────────────────────────────────┘
```

#### 监控工具选型

| 类型 | 开源方案 | 商业方案 | 适用场景 |
|------|---------|---------|---------|
| **指标监控** | Prometheus | Datadog | 时序数据 |
| **日志监控** | ELK/Loki | Splunk | 日志分析 |

### 3. 告警设计

**目的**: 及时通知问题，避免告警疲劳

#### 告警分级

| 级别 | 名称 | 响应时间 | 通知方式 | 示例 |
|------|------|---------|---------|------|
| **P0** | 紧急 | 5 分钟 | 电话 + 短信 + IM | 核心服务不可用 |
| **P1** | 严重 | 15 分钟 | 短信 + IM | 错误率>5% |
| **P2** | 高 | 1 小时 | IM | 延迟>P99 |
| **P3** | 中 | 4 小时 | IM | 资源使用>80% |
| **P4** | 低 | 24 小时 | 邮件 | 容量预警 |

#### 告警规则设计

```yaml
# Prometheus 告警规则示例
groups:
  - name: service_alerts
    rules:
      # P0: 服务不可用
      - alert: ServiceDown
        expr: up{job="order-service"} == 0
        for: 1m
        labels:
          severity: P0
        annotations:
          summary: "服务 {{ $labels.instance }} 不可用"
          
      # P1: 错误率高
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) 
              / rate(http_requests_total[5m]) > 0.05
        for: 5m
        labels:
          severity: P1
        annotations:
          summary: "错误率 {{ $value | humanizePercentage }}"
          
      # P2: 延迟高
      - alert: HighLatency
        expr: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) 
              > 1
        for: 10m
        labels:
          severity: P2
        annotations:
          summary: "P99 延迟 {{ $value }}s"
```

#### 告警管理最佳实践

| 实践 | 说明 |
|------|------|
| **告警去重** | 相同告警合并通知 |
| **告警抑制** | 维护期间暂停告警 |
| **告警升级** | 未响应自动升级 |
| **告警收敛** | 避免告警风暴 |
| **值班轮换** | 明确值班安排 |

### 4. SLO 管理

**目的**: 定义和追踪服务等级目标，平衡可靠性和迭代速度

#### SLO/SLI/SLA 定义

| 概念 | 描述 | 示例 |
|------|------|------|
| **SLI** (指标) | 服务等级指标 | 请求成功率、延迟 |
| **SLO** (目标) | 服务等级目标 | 成功率>99.9% |
| **SLA** (协议) | 服务等级协议 (对外) | 违约赔偿条款 |

#### SLO 设计模板

```markdown
## SLO 定义 - [服务名称]

### 核心 SLO
| SLI | 目标 (SLO) | 测量窗口 |
|-----|----------|---------|
| 可用性 | 99.9% | 30 天 |
| 延迟 (P99) | <500ms | 30 天 |
| 正确性 | 99.99% | 30 天 |

### 错误预算
- **月度预算**: 0.1% × 30 天 = 43.2 分钟
- **已用预算**: X 分钟
- **剩余预算**: Y 分钟
- **预算消耗率**: Z%/天

### 预算策略
| 预算剩余 | 策略 |
|---------|------|
| >50% | 正常发布 |
| 25-50% | 谨慎发布 |
| <25% | 冻结发布，专注稳定性 |

## 📋 输出规范

1. **指标可测量**: 每个指标有明确定义和采集方式
2. **告警可行动**: 告警触发后有明确处理流程
3. **SLO 可追踪**: 定期回顾 SLO 达成情况
4. **文档完整**: 监控、告警、SLO 文档齐全

---

## ⚠️ 注意事项

- ⛔ 避免告警疲劳 (过多无效告警)
- ⛔ 监控不是越多越好，关注关键指标
- ⛔ SLO 不是越高越好，平衡成本和体验
- ✅ 定期回顾和优化监控体系
