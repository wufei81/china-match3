## 🎯 技能概述

为架构师智能体提供灾备架构设计能力，包括高可用架构、多活设计、容灾策略、业务连续性计划 (BCP)。帮助系统达到 99.9%+ 可用性。

## 📊 核心功能

### 1. 高可用架构设计

**目的**: 消除单点故障，确保系统持续可用

#### 可用性等级

| 等级 | 可用性 | 年停机时间 | 适用场景 |
|------|-------|-----------|---------|
| **99%** | 2 个 9 | 87.6 小时 | 内部系统 |
| **99.9%** | 3 个 9 | 8.76 小时 | 商业系统 |
| **99.99%** | 4 个 9 | 52.6 分钟 | 关键业务 |
| **99.999%** | 5 个 9 | 5.26 分钟 | 电信/金融 |

#### 高可用策略

| 策略 | 描述 | 成本 | 复杂度 |
|------|------|------|-------|
| **冗余部署** | 多实例部署 | 低 | 低 |
| **负载均衡** | 流量分发 | 低 | 低 |
| **自动故障转移** | 自动切换 | 中 | 中 |
| **多可用区** | 跨 AZ 部署 | 中 | 中 |
| **多地域** | 跨 Region 部署 | 高 | 高 |
| **多活架构** | 多地同时服务 | 很高 | 很高 |

#### 高可用架构模板

```markdown
## 高可用架构设计 - [系统名称]

### 可用性目标
- **目标可用性**: 99.9X%
- **允许停机时间**: X 小时/年
- **RTO (恢复时间目标)**: < X 分钟
- **RPO (恢复点目标)**: < X 分钟

### 架构策略
#### 应用层高可用
- [ ] 多实例部署 (最少 2 个)
- [ ] 负载均衡 (L4/L7)
- [ ] 健康检查
- [ ] 自动故障转移
- [ ] 优雅关闭

#### 数据层高可用
- [ ] 主从复制
- [ ] 自动故障转移
- [ ] 数据备份 (每日/实时)
- [ ] 跨 AZ/Region 复制

#### 网络层高可用
- [ ] 多线路接入
- [ ] DNS 故障转移
- [ ] CDN 冗余
- [ ] 多入口网关

### 故障场景演练
| 故障场景 | 预期行为 | 实际 RTO | 实际 RPO |
|---------|---------|---------|---------|
| 单实例宕机 | 自动切换 | <30s | 0 |
| AZ 故障 | 跨 AZ 切换 | <2min | <1min |
| Region 故障 | 跨 Region 切换 | <5min | <1min |
| 数据库主库故障 | 从库升主 | <1min | <30s |
```

### 2. 多活架构设计

**目的**: 多个数据中心同时提供服务，实现容灾和负载均衡

#### 多活模式对比

| 模式 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **主备模式** | 一主多备，备用待命 | 简单、数据一致 | 备用资源闲置 | 一般业务 |
| **主从模式** | 主中心写，从中心读 | 资源利用高 | 写单点 | 读多写少 |
| **双活模式** | 两个中心同时服务 | 资源利用高 | 数据冲突 | 区域化业务 |
| **多活模式** | 多个中心同时服务 | 高可用、高性能 | 复杂度高 | 全球化业务 |

#### 数据同步策略

| 策略 | 延迟 | 一致性 | 适用场景 |
|------|------|-------|---------|
| **同步复制** | 低 | 强一致 | 金融交易 |
| **异步复制** | 中 | 最终一致 | 一般业务 |
| **半同步** | 中低 | 弱一致 | 平衡场景 |

#### 多活架构模板

```markdown
## 多活架构设计 - [系统名称]

### 部署架构
```
Region A (主) ←→ Region B (备/活)
    ↓                ↓
  AZ1 + AZ2        AZ3 + AZ4
```

### 流量调度
| 场景 | 调度策略 |
|------|---------|
| 正常 | 按用户地域就近接入 |
| 单 AZ 故障 | AZ 内负载均衡 |
| 单 Region 故障 | DNS 切流到健康 Region |

### 数据同步
- **同步方式**: [同步/异步/半同步]
- **同步延迟**: < X ms
- **冲突解决**: [时间戳/版本向量/业务规则]

### 关键挑战与方案
| 挑战 | 方案 |
|------|------|
| 数据一致性 | 分布式事务/最终一致 |
| 会话保持 | 集中式 Session/Token |
| 全局 ID | 雪花算法/号段模式 |
| 配置一致性 | 配置中心统一分发 |
```

### 3. 容灾策略与演练

**目的**: 制定容灾策略并定期演练，确保灾难发生时能有效响应

#### 容灾等级 (国标)

| 等级 | 名称 | RTO | RPO | 距离要求 |
|------|------|-----|-----|---------|
| **2 级** | 数据级容灾 | 数天 | 数天 | >10km |
| **3 级** | 应用级容灾 | 数小时 | 数分钟 | >10km |
| **4 级** | 实时容灾 | 分钟级 | 秒级 | >100km |
| **5 级** | 零丢失 | 秒级 | 0 | >300km |

#### 容灾演练计划

```markdown
## 容灾演练计划 - [系统名称]

### 演练目标
- 验证 RTO/RPO 达标
- 熟悉应急流程
- 发现潜在问题

### 演练场景
| 场景 | 频率 | 参与方 | 时长 |
|------|------|-------|------|
| 单实例故障 | 每月 | 运维 | 30min |
| 数据库主从切换 | 每季 | 运维 + 开发 | 1h |
| AZ 故障切换 | 每半年 | 全员 | 2h |
| Region 级切换 | 每年 | 全员 + 业务 | 4h |

### 演练流程
1. **准备阶段** (演练前 1 周)
   - 制定演练方案
   - 通知相关方
   - 准备回滚方案

2. **执行阶段** (演练日)
   - 按步骤执行切换
   - 记录时间点
   - 监控关键指标

3. **总结阶段** (演练后 1 天)
   - 复盘会议
   - 问题跟踪
   - 流程优化

### 演练检查清单
- [ ] 备份数据可用
- [ ] 切换脚本验证
- [ ] 监控告警正常
- [ ] 通讯渠道畅通
- [ ] 回滚方案就绪
```

### 4. 业务连续性计划 (BCP)

**目的**: 确保灾难发生时业务能持续运营

#### BCP 框架

```markdown
## 业务连续性计划 - [公司/系统名称]

### 业务影响分析 (BIA)
| 业务功能 | 重要性 | 最大中断时间 | 依赖系统 |
|---------|-------|------------|---------|
| 核心交易 | P0 | <5 分钟 | 支付、订单 |
| 用户登录 | P0 | <10 分钟 | 认证、数据库 |
| 数据查询 | P1 | <1 小时 | 搜索、缓存 |
| 报表系统 | P2 | <24 小时 | 数据仓库 |

### 应急响应组织
| 角色 | 职责 | 联系方式 |
|------|------|---------|
| 总指挥 | 决策和协调 | [姓名/电话] |
| 技术负责人 | 技术方案 | [姓名/电话] |
| 运维负责人 | 执行切换 | [姓名/电话] |
| 业务负责人 | 业务决策 | [姓名/电话] |
| 公关负责人 | 对外沟通 | [姓名/电话] |

### 应急响应流程
```
发现故障 → 初步评估 → 启动预案 → 执行切换 → 验证恢复 → 事后复盘
   ↓           ↓           ↓          ↓          ↓          ↓
 5min       10min       15min      30min      60min      24h 内
```

### 应急预案
| 预案编号 | 场景 | 触发条件 | 执行步骤 |
|---------|------|---------|---------|
| ERP-001 | 数据库故障 | 主库不可用 | [步骤链接] |
| ERP-002 | 网络故障 | 核心网络中断 | [步骤链接] |
| ERP-003 | 机房故障 | 整个 IDC 不可用 | [步骤链接] |

### 资源清单
- **备用站点**: [地址/联系人]
- **应急设备**: [清单/位置]
- **供应商支持**: [联系方式]
- **外部资源**: [云服务商/IDC]
```

## 🔧 使用示例

### 示例 1: 设计高可用架构

**输入**: "设计一个 99.99% 可用的电商系统架构"

**输出**:
```markdown
## 高可用架构 - 电商系统

### 可用性目标
- **目标**: 99.99% (年停机<53 分钟)
- **RTO**: <5 分钟
- **RPO**: <1 分钟

### 架构策略
#### 应用层
- 多可用区部署 (AZ1+AZ2)
- 自动弹性伸缩
- 负载均衡 + 健康检查

#### 数据层
- MySQL 一主两从 + MHA
- Redis Cluster 三主三从
- 每日备份 + Binlog 实时同步

#### 网络层
- 多线路 BGP 接入
- DNS 智能解析
- CDN 全站加速

### 故障场景
| 场景 | RTO | RPO |
|------|-----|-----|
| 单实例故障 | <30s | 0 |
| AZ 故障 | <2min | <30s |
| 数据库主库 | <1min | <30s |
```

## 📋 输出规范

1. **量化指标**: RTO/RPO 必须明确
2. **定期演练**: 至少每季度一次
3. **文档完整**: 预案、流程、联系人齐全
4. **持续改进**: 每次演练后复盘优化

---

## ⚠️ 注意事项

- ⛔ 灾备不是有了就行，必须定期演练
- ⛔ 避免"纸面灾备"(文档齐全但无法执行)
- ⛔ 考虑最坏情况 (多故障并发)
- ✅ 灾备投入与业务价值匹配
