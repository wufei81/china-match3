## 📊 核心功能

### 1. 威胁建模 (Threat Modeling)

**目的**: 系统识别潜在安全威胁，提前设计防护措施

#### STRIDE 威胁分类

| 威胁类型 | 描述 | 示例 | 防护措施 |
|---------|------|------|---------|
| **S**poofing (伪装) | 冒充其他用户/系统 | 盗用账号、IP 欺骗 | 身份验证、数字签名 |
| **T**ampering (篡改) | 恶意修改数据 | 修改请求参数、数据库注入 | 完整性校验、参数验证 |
| **R**epudiation (抵赖) | 否认已执行操作 | 否认发送消息、删除日志 | 审计日志、数字签名 |
| **I**nformation Disclosure (信息泄露) | 未授权访问数据 | 数据泄露、越权访问 | 加密、访问控制 |
| **D**enial of Service (拒绝服务) | 使服务不可用 | DDoS 攻击、资源耗尽 | 限流、熔断、冗余 |
| **E**levation of Privilege (权限提升) | 获取更高权限 | 越权操作、提权漏洞 | 最小权限、权限校验 |

#### 威胁建模流程

```
1. 分解系统
   - 绘制数据流图 (DFD)
   - 识别信任边界
   - 识别外部依赖

2. 识别威胁
   - 应用 STRIDE 分类
   - 头脑风暴潜在攻击
   - 参考攻击模式库 (CAPEC)

3. 评估风险
   - DREAD 评分
   - 可能性 × 影响
   - 优先级排序

4. 制定对策
   - 缓解措施
   - 接受风险
   - 转移风险
   - 避免风险
```

### DREAD 评分标准
| 维度 | 1 分 | 5 分 | 10 分 |
|------|------|------|-------|

| **Affected Users** (影响用户) | 少数用户 | 部分用户 | 所有用户 |
| **Discoverability** (可发现) | 难以发现 | 可能发现 | 公开信息 |

### 2. 安全设计原则

**目的**: 将安全原则内置于架构设计中

#### 核心安全原则

| 原则 | 描述 | 实施建议 |
|------|------|---------|
| **最小权限** | 只授予必要权限 | RBAC、细粒度授权 |
| **纵深防御** | 多层防护 | 网络 + 应用 + 数据层 |
| **默认安全** | 默认配置是安全的 | 默认关闭、默认加密 |
| **失效安全** | 失败时进入安全状态 | 默认拒绝、优雅降级 |
| **完全中介** | 所有访问都要检查 | 不信任缓存的权限 |
| **开放设计** | 安全不依赖隐蔽 | 算法公开、密钥保密 |
| **心理可接受** | 安全机制易用 | 用户友好的安全设计 |

### 3. OWASP Top 10 防护

**目的**: 针对 OWASP Top 10 漏洞提供防护方案

#### OWASP Top 10 (2021) 防护指南

| 排名 | 漏洞 | 风险等级 | 防护措施 |
|------|------|---------|---------|
| A01 | 权限控制缺失 | 🔴 严重 | RBAC、服务端校验、审计日志 |
| A02 | 加密机制缺失 | 🔴 严重 | TLS、数据加密、密钥管理 |
| A03 | 注入攻击 | 🔴 严重 | 参数化查询、输入验证、ORM |
| A04 | 不安全设计 | 🟠 高 | 威胁建模、安全设计模式 |
| A05 | 安全配置错误 | 🟠 高 | 安全基线、自动化检查 |
| A06 | 易失陷组件 | 🟠 高 | 依赖扫描、及时更新 |
| A07 | 认证失效 | 🟠 高 | MFA、会话管理、限流 |
| A08 | 软件/数据完整性 | 🟠 高 | 数字签名、完整性校验 |
| A09 | 日志与监控缺失 | 🟡 中 | 安全日志、告警、审计 |
| A10 | SSRF | 🟡 中 | URL 白名单、网络隔离 |

### 4. 零信任架构

**目的**: 基于"永不信任，始终验证"原则设计安全架构

#### 零信任核心原则

| 原则 | 描述 | 实施 |
|------|------|------|
| **明确验证** | 所有请求都要验证 | 身份 + 设备 + 上下文 |
| **最小权限** | 按需授权，及时回收 | JIT/JEA 访问 |
| **假设泄露** | 假设攻击者已在内网 | 微隔离、加密 |

#### 零信任架构模板

```markdown
## 零信任架构设计 - [项目名称]

### 身份平面
| 组件 | 功能 | 技术选型 |
|------|------|---------|
| 身份提供商 | 统一身份认证 | Keycloak/Auth0 |
| 多因素认证 | 二次验证 | TOTP/SMS/硬件密钥 |
| 会话管理 | 令牌颁发验证 | JWT/OAuth2 |

### 设备平面
| 组件 | 功能 | 技术选型 |
|------|------|---------|
| 设备注册 | 设备身份管理 | MDM 解决方案 |
| 健康检查 | 设备合规检查 | 防病毒/补丁状态 |
| 设备信任 | 动态信任评分 | 基于行为分析 |

### 网络平面
| 组件 | 功能 | 技术选型 |
|------|------|---------|
| 微隔离 | 网络分段 | 服务网格 |
| 加密通信 | 端到端加密 | mTLS |
| 访问代理 | 零信任网关 | BeyondCorp |

### 数据平面
| 组件 | 功能 | 技术选型 |
|------|------|---------|
| 数据分类 | 敏感数据识别 | DLP |
| 加密存储 | 数据加密 | 应用层加密 |
| 访问审计 | 数据访问日志 | 集中审计 |

### 实施路线图
- **Phase 1**: 身份统一 + MFA
- **Phase 2**: 设备管理 + 健康检查
- **Phase 3**: 微隔离 + mTLS
- **Phase 4**: 数据分类 + 加密
```

### 示例 2: 安全架构评审

**输入**: "评审这个 API 设计的安全性"

**输出**:
```markdown
## 安全评审 - [API 名称]

### ✅ 符合项
- 使用 HTTPS
- JWT 认证
- 输入验证

### ⚠️ 需改进
- 缺少限流防护
- 未记录审计日志
- 错误信息过于详细

### ❌ 高风险
- 直接拼接 SQL 查询
- 敏感数据明文返回

### 建议
1. 改用参数化查询
2. 添加限流中间件
3. 脱敏敏感字段
```

## 📋 输出规范

1. **威胁驱动**: 基于实际威胁设计防护
2. **分层防护**: 多层防护，纵深防御
3. **可验证**: 每项防护有验证方法
4. **持续改进**: 定期审查和更新

---

## ⚠️ 注意事项

- ⛔ 安全不是功能，是属性 (内置而非外挂)
- ⛔ 不要依赖隐蔽实现安全 (Kerckhoffs 原则)
- ⛔ 定期进行威胁建模和渗透测试
- ✅ 安全左移，设计阶段考虑
