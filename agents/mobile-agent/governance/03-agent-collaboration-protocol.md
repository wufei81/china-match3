# 智能体协作协议 (Agent Collaboration Protocol)

**适用范围:** 所有 9 个产研智能体 **执行时机:** 所有项目开发过程 **协调人:** ai-orchestrator

---

## 🎯 智能体角色定义

### 核心智能体 (9 个)

| 智能体              | 角色       | 核心职责            | 关键交付物          |
| ------------------- | ---------- | ------------------- | ------------------- |
| **ai-orchestrator** | 协调者     | 总体协调 + 最终审查 | 项目计划 + 审查报告 |
| **architect**       | 架构师     | 架构设计 + 技术选型 | 架构文档 + 技术决策 |
| **tech-lead**       | 技术负责人 | 技术规范 + 代码审查 | 代码规范 + 审查意见 |
| **product-manager** | 产品经理   | 需求分析 + 验收标准 | PRD + 用户故事      |
| **project-manager** | 项目经理   | 项目计划 + 资源协调 | 里程碑 + 进度报告   |
| **dev-engineer**    | 开发工程师 | 代码实现 + 单元测试 | 源代码 + 测试代码   |
| **qa-engineer**     | 测试工程师 | 测试计划 + 质量保障 | 测试用例 + 测试报告 |
| **devops-engineer** | 运维工程师 | 部署配置 + 监控告警 | CI/CD + 监控面板    |
| **ui-ux-designer**  | 设计师     | UI 设计 + 用户体验  | 设计稿 + 交互原型   |

---

## 📋 协作流程

### 标准开发流程

```
需求阶段
    ↓
product-manager 创建 PRD
    ↓
architect 设计架构
    ↓
tech-lead 制定规范
    ↓
    ├─ dev-engineer 实现代码
    ├─ qa-engineer 编写测试
    └─ devops-engineer 配置部署
    ↓
ui-ux-designer 优化体验
    ↓
project-manager 跟踪进度
    ↓
ai-orchestrator 最终审查
    ↓
✅ 交付
```

## 🔧 沟通机制

### 1. 每日站会 (Daily Standup)

**时间:** 每天上午 9:00 (自动触发) **时长:** 15 分钟 **参与:** 所有智能体 **内容:**

- 昨天完成了什么
- 今天计划做什么
- 遇到什么阻碍

**输出:** 站会纪要 (自动记录)

### 2. 周例会 (Weekly Review)

**时间:** 每周一上午 10:00 **时长:** 30 分钟 **参与:** ai-orchestrator + 各智能体代表 **内容:**

- 上周进度回顾
- 本周计划确认
- 风险问题讨论

**输出:** 周报 + 风险清单

### 3. 里程碑评审 (Milestone Review)

**时间:** 每个里程碑结束时 **时长:** 60 分钟 **参与:** 所有智能体 **内容:**

- 里程碑成果展示
- 质量审查报告
- 下阶段计划

**输出:** 里程碑报告 + 验收文档

## 📝 代码修改协议

### 修改前检查清单

```yaml
# 智能体修改代码前必须检查
- [ ] 是否影响现有功能
- [ ] 是否需要同步更新其他模块
- [ ] 是否需要更新文档
- [ ] 是否需要添加测试
- [ ] 是否符合代码规范
```

### 修改后验证清单

```yaml
# 智能体修改代码后必须验证
- [ ] 语法检查通过
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 代码格式化
- [ ] 导入排序正确
```

### 提交规范

```
<type>(<scope>): <subject>

<body>

<footer>
```

**type 类型:**

- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `style`: 代码格式
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具

**示例:**

```
feat(api): 添加用户管理 API

- 实现用户 CRUD 操作
- 添加权限验证
- 编写单元测试

Closes: #123
```

## 🚨 冲突解决机制

### 冲突类型与处理

| 冲突类型         | 处理方式   | 决策人                |
| ---------------- | ---------- | --------------------- |
| **技术选型冲突** | 技术评审会 | architect + tech-lead |
| **需求变更冲突** | 需求评审会 | product-manager       |
| **资源分配冲突** | 资源协调会 | project-manager       |
| **代码质量冲突** | 代码审查会 | tech-lead             |
| **架构设计冲突** | 架构评审会 | architect             |
| **无法达成共识** | 升级决策   | ai-orchestrator       |

### 升级流程

```
智能体间协商 (24 小时)
    ↓
无法达成共识
    ↓
相关智能体评审会 (48 小时)
    ↓
仍无法达成共识
    ↓
ai-orchestrator 最终决策
    ↓
✅ 执行决策
```

## 🎯 智能体职责矩阵 (RACI)

| 活动     | ai-orchestrator | architect | tech-lead | dev-engineer | qa-engineer |
| -------- | --------------- | --------- | --------- | ------------ | ----------- |
| 架构设计 | A               | R         | C         | I            | I           |
| 代码实现 | I               | C         | C         | R            | I           |
| 代码审查 | A               | C         | R         | I            | C           |
| 测试编写 | I               | I         | C         | C            | R           |
| 部署配置 | I               | I         | C         | C            | I           |
| 需求分析 | C               | I         | C         | I            | I           |
| 项目计划 | C               | I         | I         | I            | I           |
| 最终交付 | R               | C         | C         | C            | C           |

**图例:**

- R = Responsible (负责执行)
- A = Accountable (最终负责)
- C = Consulted (咨询)
- I = Informed (通知)

## 🚀 持续改进

### 回顾会议 (Retrospective)

- 什么做得好 (Keep)
- 什么需要改进 (Improve)

- 什么需要开始 (Start)

**输出:** 改进行动清单

- 最佳实践文档
- 问题案例库
