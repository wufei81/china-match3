# Stakeholder Management Skill - 利益相关者管理技能

**版本**: 1.0 **创建日期**: 2026-03-26 **触发词**: "利益相关者", "stakeholder", "期望管理", "风险沟通", "向上管理",
"跨部门协作"

---

## 🎯 技能概述

为产品经理智能体提供利益相关者管理能力，包括 stakeholder
mapping、期望管理模板、风险沟通策略。帮助从"任务执行者"升级为"组织协调者"。

---

## 📊 核心功能

### 1. Stakeholder Mapping (利益相关者地图)

**目的**: 识别所有相关方，理解其影响力和利益诉求，制定沟通策略

#### 权力 - 利益矩阵

```
                    权力 (Power)
              高 ←───────────→ 低
            ┌─────────┬─────────┐
         高 │ 重点管理 │ 保持满意 │
    利       │ (Manage)│ (Satisfy)│
    益   ───┼─────────┼─────────┤
    (      │ 随时沟通 │ 最小努力 │
    Interest)│ (Inform)│ (Monitor)│
            └─────────┴─────────┘
```

#### Stakeholder Mapping 模板

```markdown
## 利益相关者地图 - [项目/产品名称]

### 一、利益相关者清单

| 姓名/角色 | 部门   | 职位   | 影响力   | 利益度   | 态度           |
| --------- | ------ | ------ | -------- | -------- | -------------- |
| [姓名]    | [部门] | [职位] | 高/中/低 | 高/中/低 | 支持/中立/反对 |

### 二、权力 - 利益矩阵

#### 重点管理 (高权力 - 高利益)

| 姓名   | 诉求       | 沟通策略               |
| ------ | ---------- | ---------------------- |
| [姓名] | [核心诉求] | 每周 1v1，深度参与决策 |

#### 保持满意 (高权力 - 低利益)

| 姓名   | 诉求       | 沟通策略                   |
| ------ | ---------- | -------------------------- |
| [姓名] | [核心诉求] | 关键节点汇报，避免细节打扰 |

#### 随时沟通 (低权力 - 高利益)

| 姓名   | 诉求       | 沟通策略           |
| ------ | ---------- | ------------------ |
| [姓名] | [核心诉求] | 定期同步，收集反馈 |

#### 最小努力 (低权力 - 低利益)

| 姓名   | 诉求       | 沟通策略           |
| ------ | ---------- | ------------------ |
| [姓名] | [核心诉求] | 邮件同步，按需沟通 |

### 三、关键利益相关者分析

#### [姓名/角色]

- **职位**: [具体职位]
- **影响力来源**: [决策权/预算/资源/信息]

- **潜在顾虑**: [可能反对的原因]
- **支持策略**: [如何争取支持]
- **沟通偏好**: [会议/邮件/即时消息]

### 四、沟通计划

| 利益相关者 | 沟通方式 | 频率 | 负责人   | 下次沟通 |
| ---------- | -------- | ---- | -------- | -------- |
| [姓名]     | 1v1 会议 | 每周 | [负责人] | [日期]   |
| [姓名]     | 邮件同步 | 双周 | [负责人] | [日期]   |
| [全体]     | 项目例会 | 每周 | [负责人] | [日期]   |
```

#### 常见利益相关者类型

| 角色           | 典型诉求                | 沟通要点                   |
| -------------- | ----------------------- | -------------------------- |
| **高管/老板**  | ROI、战略对齐、风险控制 | 讲结果、讲价值、讲风险     |
| **技术负责人** | 技术可行性、架构合理性  | 讲清楚需求边界和技术约束   |
| **设计师**     | 用户体验、设计完整性    | 给足设计空间，尊重专业     |
| **开发团队**   | 需求清晰、排期合理      | 需求零歧义，预留 buffer    |
| **测试团队**   | 测试时间、用例清晰      | 提前同步 AC，预留测试时间  |
| **运营团队**   | 上线时间、培训支持      | 提前同步上线计划，提供培训 |
| **销售团队**   | 卖点清晰、客户案例      | 提炼卖点，提供销售工具     |
| **客服团队**   | 问题预案、FAQ           | 提前培训，准备应答话术     |

### 2. 期望管理模板

**目的**: 对齐各方期望，避免期望落差导致的冲突和失望

#### 期望管理框架

```markdown
## 期望管理 - [项目/产品名称]

### 一、期望对齐表

| 利益相关者 | 期望       | 现实       | 差距       | 管理策略   |
| ---------- | ---------- | ---------- | ---------- | ---------- |
| [姓名]     | [他的期望] | [实际情况] | [差距描述] | [如何管理] |

### 二、承诺管理

#### 已承诺事项

| 承诺内容 | 承诺对象 | 承诺日期 | 交付日期 | 状态          |
| -------- | -------- | -------- | -------- | ------------- |
| [承诺]   | [对象]   | [日期]   | [日期]   | 进行中/已完成 |

#### 无法承诺事项

| 请求   | 请求对象 | 无法承诺原因 | 替代方案   |
| ------ | -------- | ------------ | ---------- |
| [请求] | [对象]   | [原因]       | [替代方案] |

### 三、期望管理话术

#### 当被要求不可能完成的任务时

> "我理解这个需求的重要性。基于当前资源，我们可以在 [日期] 交付 [范围]。如果要提前，我们需要
> [增加资源/减少范围/调整优先级]。您看哪个方案更合适？"

#### 当需求频繁变更时

> "我理解业务在快速变化。为了平衡灵活性和交付效率，建议：1) 核心需求锁定后不变 2) 非核心需求进入 backlog 3) 每周评审一次优先级。您觉得如何？"

#### 当被质疑进度时

> "当前进度 [X%]，比计划 [提前/延迟] [Y 天]。主要原因是 [原因]。我们的应对方案是 [方案]。预计 [日期] 可以 [里程碑]。"

### 四、期望管理检查清单

- [ ] 所有关键利益相关者的期望已记录
- [ ] 不切实际的期望已被识别并沟通
- [ ] 承诺事项有明确交付日期
- [ ] 无法承诺的事项已提供替代方案
- [ ] 定期同步进展，避免惊喜
```

#### 期望管理原则

| 原则                            | 说明                   | 示例                               |
| ------------------------------- | ---------------------- | ---------------------------------- |
| **Under-promise, Over-deliver** | 承诺保守，交付超额     | 承诺 2 周，实际 10 天交付          |
| **透明沟通**                    | 坏消息早说，好消息晚说 | 风险提前暴露，不隐瞒问题           |
| **管理变更**                    | 变更有成本，需权衡     | "可以加功能，但要延期或砍其他功能" |
| **记录在案**                    | 重要承诺书面确认       | 会议纪要、邮件确认                 |

### 3. 风险沟通策略

**目的**: 提前识别和沟通风险，避免问题爆发时的信任危机

#### 风险沟通框架

```markdown
## 风险沟通 - [项目/产品名称]

### 一、风险登记册

| 风险       | 概率     | 影响     | 风险值 | 责任人 | 应对策略 |
| ---------- | -------- | -------- | ------ | ------ | -------- |
| [风险描述] | 高/中/低 | 高/中/低 | P×I    | [姓名] | [策略]   |

### 二、风险沟通计划

#### 高风险 (立即沟通)

| 风险   | 沟通对象 | 沟通方式 | 沟通时间 |
| ------ | -------- | -------- | -------- |
| [风险] | [对象]   | [方式]   | [时间]   |

#### 中风险 (定期同步)

| 风险   | 沟通对象 | 沟通方式 | 频率   |
| ------ | -------- | -------- | ------ |
| [风险] | [对象]   | [方式]   | [频率] |

#### 低风险 (记录跟踪)

| 风险   | 跟踪方式 | 触发条件         |
| ------ | -------- | ---------------- |
| [风险] | [方式]   | [什么情况下升级] |

### 三、风险沟通话术

#### 风险预警

> "我想提前同步一个潜在风险：[风险描述]。目前概率 [X%]，如果发生会影响 [影响描述]。我们建议
> [应对方案]。想听听您的意见。"

#### 风险升级

> "[风险] 已经发生/概率显著增加。影响范围：[描述]。建议立即采取 [行动]。需要您支持 [具体支持]。"

#### 风险解除

> "[风险] 已解除。原因：[原因]。后续我们会 [预防措施]，避免类似问题。"

### 四、常见风险类型

| 风险类型     | 示例                 | 应对策略                      |
| ------------ | -------------------- | ----------------------------- |
| **进度风险** | 开发延期、依赖方延迟 | 预留 buffer，提前识别关键路径 |
| **范围风险** | 需求变更、范围蔓延   | 变更控制流程，优先级排序      |
| **技术风险** | 技术难点、性能问题   | 技术预研，POC 验证            |
| **资源风险** | 人力不足、关键人离职 | 资源备份，知识文档化          |
| **外部风险** | 政策变化、第三方依赖 | 监控外部变化，准备备选方案    |
```

#### 风险沟通原则

| 原则         | 说明                       |
| ------------ | -------------------------- |
| **尽早沟通** | 风险越早暴露，解决成本越低 |
| **带着方案** | 不要只抛问题，带上建议方案 |
| **分级沟通** | 不同级别风险，不同沟通策略 |
| **持续跟踪** | 风险状态定期更新           |

### 示例 2: 风险沟通

**输入**: "项目可能延期 2 周，怎么跟老板沟通"

**输出**:

```markdown
## 风险沟通建议

### 沟通话术

> "张总，我想提前同步一个潜在风险：由于 [具体原因，如第三方 API 延期]，项目可能延期 2 周。目前概率 80%，如果发生会影响
> [具体影响]。

> 我们有三个应对方案：
>
> 1. **方案 A**: 按原计划上线核心功能，非核心功能延后 (推荐)
> 2. **方案 B**: 整体延期 2 周，保证完整功能
> 3. **方案 C**: 增加 2 名开发，赶原计划 (成本增加 X 万)

> 我建议方案 A，理由是 [理由]。想听听您的意见。"

### 沟通要点

- ✅ 提前沟通，不隐瞒
- ✅ 带上方案，不只是问题
- ✅ 给出推荐，说明理由
- ✅ 让老板做选择题，不是问答题
```

---

## 📋 输出规范

1. **具体到人**: Stakeholder 地图要具体到姓名和职位
2. **可执行**: 沟通策略要具体到方式和频率
3. **提前预警**: 风险沟通要提前，不要等问题爆发
4. **持续更新**: 定期回顾和更新利益相关者状态

---

## ⚠️ 注意事项

- ⛔ 不要假设利益相关者的诉求，要主动沟通确认
- ⛔ 不要只报喜不报忧，坏消息要早说
- ⛔ 不要口头承诺，重要承诺要书面确认
- ✅ 定期回顾 Stakeholder 地图，人员会变动
