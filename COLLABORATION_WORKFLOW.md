## 一、核心原则

### 1.1 交付优先级 (不可违背)

```
1. 实现真实功能 → 2. 能运行 → 3. 界面美观
```

**严禁**:
- ❌ 占位符、空实现、Mock 替代真实 API/DB
- ❌ "代码无报错"作为完成依据
- ❌ 为美观牺牲功能实现

### 1.2 文档先行 (无文档不开发)

```
BRD → MRD → PRD → FSD → 架构 → API 契约 → 开发 → 测试 → 发布
 ↑      ↑      ↑      ↑       ↑         ↑
 └────────────── 任一环节缺失 = 停工 ─────────────┘
```

### 1.3 多模型协同 (避免锚定效应)

**核心方法**: 独立思考 → 公开辩论 → 融合收敛

## 二、多产品经理协同机制

### 2.1 三 PM 分工

| PM | 模型 | 专注领域 | 主责文档 |
|----|------|----------|----------|
| `product-manager-qwen` | 通义千问 | 需求完整性、业务逻辑、优先级 | PRD、BRD |
| `product-manager-glm` | 智谱 GLM | 用户体验、场景覆盖、交互流程 | FSD、MRD |
| `product-manager-kimi` | Kimi | 商业模式、ROI、风险评估 | BRD 商业分析 |

### 2.2 三 PM 评审流程

```
Phase 1: 独立思考 (隔离上下文)
┌────────────────────────────────────────────────────┐
│  PM-qwen: 撰写 PRD 草案 (需求完整性视角)             │
│  PM-glm:  独立撰写 UX 评估 (用户体验视角)            │
│  PM-kimi: 独立撰写商业评估 (ROI/风险视角)          │
└────────────────────────────────────────────────────┘
                    ↓
Phase 2: 公开辩论 (交换方案 + 交叉评审)
┌────────────────────────────────────────────────────┐
│  - 每个 PM 必须说出"我不同意 X 因为..."               │
│  - 允许被说服、修改立场                            │
│  - 记录所有分歧点和解决理由                        │
└────────────────────────────────────────────────────┘
                    ↓
Phase 3: 融合收敛
┌────────────────────────────────────────────────────┐
│  - 提取各方案最优部分                              │
│  - 解决冲突点 (明确记录决策理由)                   │
│  - 输出"任何单一 PM 都想不出"的方案                 │
└────────────────────────────────────────────────────┘
                    ↓
Phase 4: 签字冻结
┌────────────────────────────────────────────────────┐
│  - 三 PM 共同签字                                   │
│  - tech-leader 批准技术可行性                       │
│  - 文档存入 docs/ + 版本号                          │
└────────────────────────────────────────────────────┘
```

## 四、双研发工程师协同机制

### 4.1 双 Dev 分工策略

**按模块分工 (推荐)**:
```
┌─────────────────────────────────────────────────────┐
│  模块 A (用户认证) → dev-engineer-qwen              │
│  模块 B (消息系统) → dev-engineer-glm               │
│  共享层 (DB/Cache) → 共同设计，轮流实现             │
└─────────────────────────────────────────────────────┘
```

**按前后端分工 (备选)**:
```
┌─────────────────────────────────────────────────────┐
│  后端 API → dev-engineer-qwen                        │
│  前端页面 → dev-engineer-glm                         │
│  接口契约 → architect 定义，双方共同评审              │
└─────────────────────────────────────────────────────┘
```

### 4.2 双 Dev 协同流程

```
Phase 0: 需求复述 (共同完成)
┌────────────────────────────────────────────────────┐
│  - 各 Dev 独立复述需求 (用自己的话)                  │
│  - 对比复述差异，识别理解偏差                      │
│  - 与 product-manager 对齐，签字确认               │
└────────────────────────────────────────────────────┘
                    ↓
Phase 1: 接口对齐 (共同设计)
┌────────────────────────────────────────────────────┐
│  - 共同评审 architect 的 API 契约                    │
│  - 确认方法名/参数/返回类型一致                    │
│  - 识别接口依赖，定义 Mock 数据 (仅用于并行开发)     │
└────────────────────────────────────────────────────┘
                    ↓
Phase 2: 并行实现 (隔离上下文)
┌────────────────────────────────────────────────────┐
│  - Dev-qwen: 实现模块 A (独立 session)              │
│  - Dev-glm:  实现模块 B (独立 session)              │
│  - 每日同步接口变更 (避免脱节)                     │
└────────────────────────────────────────────────────┘
                    ↓
Phase 3: 交叉评审 (Code Review)
┌────────────────────────────────────────────────────┐
│  - Dev-qwen 评审 Dev-glm 的代码                      │
│  - Dev-glm  评审 Dev-qwen 的代码                     │
│  - 记录问题 → 修复 → 重新评审                      │
└────────────────────────────────────────────────────┘
                    ↓
Phase 4: 集成自验 (共同完成)
┌────────────────────────────────────────────────────┐
│  - 共同运行 E2E 测试                                 │
│  - 共同填写 AC_TRACE_MATRIX.md                      │
│  - 共同录制演示视频                                │
│  - 共同签字交付 QA                                  │
└────────────────────────────────────────────────────┘
```

```python
# 伪代码：ai-orchestrator 调度逻辑

def schedule_task(task):
    # 1. 判断任务类型
    if task.type == "PRD_Review":
        # 三 PM 并行启动 (隔离上下文)
        spawn_agent("product-manager-qwen", task.prd, session="isolated-pm1")
        spawn_agent("product-manager-glm", task.prd, session="isolated-pm2")
        spawn_agent("product-manager-kimi", task.prd, session="isolated-pm3")
        # 等待全部完成 → 启动辩论 → 融合收敛
        
    elif task.type == "UI_Design":
        # 双 UI 并行启动
        spawn_agent("ui-ux-designer-qwen", task.fsd, session="isolated-ui1")
        spawn_agent("ui-ux-designer-kimi", task.fsd, session="isolated-ui2")
        # 等待完成 → 对齐评审 → 合并交付
        
    elif task.type == "Development":
        # 双 Dev 并行启动 (按模块分工)
        spawn_agent("dev-engineer-qwen", task.module_a, session="isolated-dev1")
        spawn_agent("dev-engineer-glm", task.module_b, session="isolated-dev2")
        # 等待完成 → 交叉评审 → 集成自验
        
    elif task.type == "Architecture":
        # 架构师独立设计 → tech-leader 评审
        spawn_agent("architect", task.fsd, session="isolated-arch")
        # 等待完成 → tech-leader 审查 → 签字
        
    # 2. 跟踪进度 (阶段变化时汇报，非机械刷屏)
    # 3. 门禁检查 (无文档不进入下一阶段)
    # 4. 交付验收 (E2E 通过才签署发布)
```

## 八、超时处理策略

### 8.1 T-shirt 体量评估

| 体量 | 单次子任务上限 | 超时处理 |
|------|---------------|----------|
| **S** (小) | 15-25 min | 拆更细或降级为 T0 |
| **M** (中) | 45-60 min | 拆里程碑 (接口→核心→边界) |
| **L** (大) | 分多个子任务，每个≤60min | 禁止单次等 2h+，改并行/排队 |

### 8.2 超时处理流程

```
检测到超时 → 分析原因 → 选择策略

原因 1: 任务过大
→ 拆里程碑 (接口→核心逻辑→边界处理)
→ spawn 新子任务 (每个≤60min)

原因 2: 依赖阻塞
→ 识别依赖 (等待谁？)
→ 通知依赖方 + 调整优先级
→ 临时降级 (先做不依赖部分)

原因 3: 理解偏差
→ 暂停任务
→ 重新对齐需求 (product-manager 参与)
→ 更新任务描述 → 重启

原因 4: 技术难点
→ 升级 tech-leader
→ 启动架构师协助
→ 调整方案 (简化/替代方案)
```

## 九、沟通与汇报机制

### 9.1 汇报时机

**必须汇报**:
- ✅ 阶段变化 (完成一个 Gate)
- ✅ 阻塞解除 (依赖方完成)
- ✅ 里程碑完成 (L 任务拆分的子任务完成)
- ✅ 风险发现 (P0/P1 问题)

**无需汇报**:
- ❌ 机械进度更新 (每 2-3 分钟)
- ❌ 正常工作中 (无阶段变化)
- ❌ 等待依赖时 (已汇报过)

### 9.2 汇报格式

```markdown
## 📊 进度汇报 - [任务名称]

**当前阶段**: [Gate X / Phase X]
**状态**: 🟢 正常 / 🟡 风险 / 🔴 阻塞

**本阶段完成**:
- ✅ [完成项 1]
- ✅ [完成项 2]

**下一阶段**:
- 🔜 [下一步计划]

**需要支持**:
- ⚠️ [如有阻塞，说明需要谁协助]

**预计完成**: [时间估算]
```
