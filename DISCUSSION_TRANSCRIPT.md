## Phase 3: 优化方案共识

### 共识 1: 有限并行工作流

```
Phase 1: 需求 + 架构 (并行)
- PM 写 PRD
- Arch 同时看需求，构思方案
- UI 同时看需求，构思设计

Phase 2: 设计 + 技术评审 (并行)
- UI 完成设计稿
- Arch 完成 API Schema
- Tech Leader 评审架构

Phase 3: 开发 + 测试准备 (并行)
- Dev 开发
- QA 基于 PRD 写测试用例
- PM 追踪进度

Phase 4: 测试 + 部署准备 (并行)
- Dev 完成前 1 天，Ops 准备部署
- Dev 完成后，QA 立即测试
- 测试通过，Ops 部署
```

---

### 共识 2: STATE.md 使用规范

**更新频率**: 
- 任务状态变更时立即更新
- 每 2 小时至少更新一次进度

**更新内容**:
- 任务进度（%）
- 当前状态（进行中/已完成/阻塞）
- 变更内容（API/设计/需求）
- 阻塞问题

**通知机制**:
- 变更影响的相关方自动@
- 要求 4h 内确认收到

---

### 共识 3: 变更通知流程

```
变更发生 → 更新 STATE.md → 自动通知相关方 → 相关方确认 → 同步更新

通知规则:
- PRD 变更 → Arch, UI, QA (24h 确认)
- API 变更 → Dev, QA (4h 确认)
- 设计变更 → Dev, QA (24h 确认)
- 测试失败 → Dev, PM (立即)
- 阻塞发现 → PM, Tech (1h 确认)
```

---

### 共识 4: 质量门禁左移

**开发前评审** (Tech Leader 负责):
- [ ] PRD 已读
- [ ] API Schema 已确认
- [ ] 技术方案清晰
- [ ] 风险已识别
- [ ] Tech Leader 签字

**开发中自检查** (DevEngineer 负责):
- [ ] 代码能编译
- [ ] 单元测试通过
- [ ] 没有硬编码密钥
- [ ] 错误处理完整

**交付前自验证** (DevEngineer 负责):
- [ ] npm run lint 通过
- [ ] npm run test 通过 (≥80%)
- [ ] npm run build 成功
- [ ] 健康检查通过
- [ ] 关键 API 测试通过
- [ ] VALIDATION_LOG.md 填写
