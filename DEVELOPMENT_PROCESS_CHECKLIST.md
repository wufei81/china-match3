# 📋 DEVELOPMENT_PROCESS_CHECKLIST.md

> **快速参考卡片** - 所有智能体在开发任务时必须遵循的 6 阶段流程
>
> **适用智能体**: `dev-engineer` (执行), `tech-leader` (监督), `qa-engineer` (验证), `ai-orchestrator` (追踪)

---

## 🎯 6 阶段开发流程 (MANDATORY)

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 0: 需求澄清 (5-10 min)                                │
│  ⛔ 禁止开始写代码！先明确 5 个关键点                            │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: 定位入口 + 建立上下文 (10-20 min)                   │
│  🔍 找入口层 → 追核心层 → 识别消费方 → 记录关键事实            │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 2: 方案决策 (5-10 min)                                │
│  小改动→直接做 | 中改动→确认方案 | 大改动→设计文档             │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 3: 微设计 (5-10 min)                                  │
│  拆成子任务 A-E: 接口→核心→错误→日志→测试                      │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 4: 编码 (最小可运行增量)                               │
│  主路径→边界→一致性→清理 (按顺序，别跳步)                       │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 5: 验证 (10-20 min)                                   │
│  静态→单元→运行→回归 (4 层验证)                                 │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  Phase 6: 对齐 (按需)                                        │
│  有不确定性→停止→等用户确认                                   │
└─────────────────────────────────────────────────────────────┘
```

## ✅ Phase 0: 需求澄清 (5 个关键点)

**⛔ 禁止在未完成此阶段前开始写代码！**

| #   | 问题                                                | 示例回答                                            |
| --- | --------------------------------------------------- | --------------------------------------------------- |
| 1   | **功能目标**：新增/修改/删除什么行为？              | "新增用户登录 API，支持邮箱密码登录"                |
| 2   | **触发方式**：从哪里调用？谁会用到？                | "POST /api/auth/login，前端登录页面调用"            |
| 3   | **输入输出**：参数、返回结构、数据格式？            | "输入：{email, password} 输出：{token, expiresIn}"  |
| 4   | **失败行为**：异常/空结果/超时/权限不足时怎么表现？ | "401 + {error: 'INVALID_PASSWORD', message: '...'}" |
| 5   | **质量要求**：性能、可读性、兼容性、测试？          | "P95 <100ms，加单元测试，向后兼容"                  |

**记录位置**: `TASK_TRACKER.md` → "Requirements Memo" 区块

**如果任何点不清楚**:

- ⛔ **停止**，问用户
- ⛔ **不要**假设或猜测

## ✅ Phase 1: 定位入口 + 建立上下文

**目标**: 知道"改哪里、怎么连上现有系统"

### Step 1.1: 找入口

- [ ] **入口层**: API handler / CLI 入口 / 能力注册点
- [ ] **核心层**: 真正执行逻辑的函数/类
- [ ] **消费方**: 上游如何拿结果，下游如何使用
- [ ] **接口契约**: 确保一致性

### Step 1.2: 识别已有抽象

- [ ] 已有同类实现？(可复用，别重写)
- [ ] 统一错误类型？
- [ ] 日志约定？
- [ ] 配置读取方式？
- [ ] 数据结构约定？
- [ ] runner/adapter 模式？(如 `sql_runner` base)

### Step 1.3: 记录关键事实 (Context Memo)

**模板**:

```markdown
## Context Memo: [任务名称]

**入口**: `src/api/auth/login.ts` → `src/services/auth.service.ts` **核心函数**: `login()`, `generateJWT()`,
`verifyPassword()` **消费方**: 前端登录页面、session 中间件 **错误模式**: 抛 `AuthError`，错误码：`USER_NOT_FOUND`,
`INVALID_PASSWORD` **配置模式**: `os.getenv()` 读密钥，`config.py` 读应用设置 **已有抽象**: `BaseService`
类处理日志、指标、错误包装
```

**记录位置**: `TASK_TRACKER.md` 或 `CODEBASE_INDEX.md`

## ✅ Phase 2: 方案决策

**先定策略，再落地**

### 改动分类

| 类型       | 特征                             | 策略                            |
| ---------- | -------------------------------- | ------------------------------- |
| **小改动** | Bug 修复、微调、无行为变化       | 直接实现，最小改动              |
| **中改动** | 接口变化、错误语义、数据格式变化 | **先口头/文字确认方案**，再实现 |
| **大改动** | 新模块、重构、数据流变化         | **设计文档**，获批后实现        |

### 坑点检查

- ⚠️ 返回结构会破坏兼容吗？
- ⚠️ 异常会被捕获并映射到统一错误层吗？
- ⚠️ 有异步/超时/重试逻辑要配合吗？
- ⚠️ 边界条件与旧行为一致吗？

**中/大改动**:

- ✅ 在 `TASK_TRACKER.md` 写设计摘要
- ✅ 等用户确认后再写代码

## ✅ Phase 3: 微设计

**拆成可控子任务**

| 子任务           | 描述                                 | 验收标准         |
| ---------------- | ------------------------------------ | ---------------- |
| **A: 接口/签名** | 函数签名、参数、类型、默认值         | 类型检查通过     |
| **B: 核心逻辑**  | 主路径实现                           | 主路径跑通       |
| **C: 错误处理**  | 边界条件、异常、空值、超时、资源释放 | 所有错误分支覆盖 |
| **D: 可观测性**  | 日志、指标、错误消息                 | 日志可定位问题   |
| **E: 测试/验证** | 单元测试、集成测试、验证脚本         | 测试通过         |

**记录位置**: `TASK_TRACKER.md` → "Subtasks" 区块

## ✅ Phase 4: 编码 (最小可运行增量)

**⛔ 禁止一次性写完所有代码再赌运气！**

### Stage 4.1: 主路径优先

- [ ] 让**主路径**先跑起来
- [ ] 确保能被**调用到**(如有注册机制)
- [ ] 验证基本结果正确
- [ ] **立即测试**——别等所有代码写完

### Stage 4.2: 边界与失败分支

- [ ] 加异常处理 (空值、类型不匹配、数据验证失败)
- [ ] 对齐**系统错误语义** (抛异常 vs 返回错误对象)
- [ ] 处理超时、重试、资源释放

### Stage 4.3: 一致性整理

- [ ] **类型一致**: 返回结构统一、字段命名一致
- [ ] **行为一致**: 与同类能力同风格
- [ ] **依赖一致**: 用已有基类/工具 (别绕过 `sql_runner` base)

### Stage 4.4: 清理与小优化 (最后！)

- [ ] 抽公共函数 (减少重复)
- [ ] 复杂逻辑加简短注释
- [ ] ⛔ **禁止**大范围重构 (除非用户明确要求)

## ✅ Phase 5: 验证

**不是"跑通"，而是"验证关键点没坏"**

| 优先级 | 验证层    | 检查内容                             |
| ------ | --------- | ------------------------------------ |
| **P0** | 静态层    | 类型检查、linter                     |
| **P1** | 单元/集成 | 更新/添加测试                        |
| **P2** | 运行层    | 最小可复现路径 (典型输入 + 边界输入) |
| **P3** | 回归层    | 确保旧行为没被意外改变               |

### 如果环境跑不了测试

提供**最小验证清单**给用户：

```markdown
## 用户验证清单

1. 运行：`curl -X POST http://localhost:3000/api/auth/login -d '{...}'` 预期：`{"token":"...","expiresIn":3600}`

2. 运行：`curl -X POST ... -d '{...}'` (错误密码) 预期：`{"error":"INVALID_PASSWORD"}`

3. 运行：`npm run test -- auth.test.ts` 预期：所有测试通过
```

## ✅ Phase 6: 对齐

**出现不确定性时，卡住等用户确认**

### 停止场景

| 场景               | 为什么停                 |
| ------------------ | ------------------------ |
| **需求歧义**       | 输出格式/错误语义不清楚  |
| **兼容性策略不明** | 是否允许返回结构变化     |
| **安全/敏感配置**  | `.env`、密钥、权限涉及   |
| **需要样例数据**   | 用户需提供测试数据       |
| **设计决策需要**   | 多个有效方案，需用户偏好 |

### 如何升级

1. 清楚说明不确定性
2. 提供 2-3 个选项 + 权衡
3. 等用户决定后再继续

## 🚨 违规处理

| 违规行为                    | 发现者      | 处理           |
| --------------------------- | ----------- | -------------- |
| Phase 4 在 Phase 0-3 前开始 | tech-leader | 打回，要求补全 |
| 无 Context Memo             | tech-leader | 打回，要求记录 |
| 中/大改动无用户确认         | qa-engineer | 打回，要求确认 |
| 一次性提交所有代码          | tech-leader | 打回，要求增量 |
| 无 4 层验证                 | qa-engineer | 拒绝交付       |
